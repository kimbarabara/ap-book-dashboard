# 교재 대시보드 — 관리자 화면 분리 요구사항

## 1. 배경 / 목적

현재 관리자에게는 탭이 7개 노출됨:
`교재 낱개 검색 | 반별 교재 조회 | 과목별 교재 조회 | 미등록 교재 요청함 | 권한 관리 | 감사 로그 | 접속 현황`

조회용 탭 3개와 관리용 탭 4개가 한 줄에 섞여 있어 복잡함.
→ **관리용 4개를 별도의 "관리자 화면"으로 분리**하고, 메인 화면에는 조회 탭 3개만 남긴다.

## 2. 구현 방식 (확정): 같은 페이지 내 화면 전환 (SPA 방식)

- **새 탭/별도 URL로 열지 않는다.** 같은 웹앱 페이지 안에서 메인 화면 ↔ 관리자 화면을 전환한다.
- 이유: Apps Script 새 탭은 매번 전체 재로딩(1~2초)이 발생하고, URL 직접 접근에 대한 별도 권한 방어가 필요해짐. 화면 전환 방식이 더 단순하고 즉시 반응함.
- 외부 문서인 "DB 스프레드시트 열기"만 기존대로 새 탭으로 연다. (외부 문서 = 새 탭, 앱 내부 기능 = 앱 안에서 전환)

## 3. UI 스펙

### 3-1. 진입점: 톱니바퀴 아이콘 ⚙
- 위치: **"DB 스프레드시트 열기" 버튼 바로 옆** (헤더 영역).
- **관리자 권한에게만 렌더링.** 직원·뷰어에게는 아이콘 자체가 서버 응답에 포함되지 않아야 함 (CSS 숨김 금지).
- 클릭 시 → 관리자 화면으로 전환.

### 3-2. 메인 화면 (전환 전)
- 탭 3개만 노출: `교재 낱개 검색 | 반별 교재 조회 | 과목별 교재 조회`
- 관리자/직원/뷰어 모두 동일한 탭 구성 (권한별 차이는 탭이 아니라 탭 내부 데이터·기능에서 발생).

### 3-3. 관리자 화면 (전환 후)
- 서브탭 4개: `미등록 교재 요청함 | 권한 관리 | 감사 로그 | 접속 현황`
- 좌측 상단에 **"← 대시보드로"** 버튼 → 클릭 시 메인 화면으로 복귀.
- 복귀 시 메인 화면의 이전 상태(선택했던 탭, 조회 조건, 결과) 유지되면 좋음 (화면을 DOM에서 제거하지 말고 show/hide 방식 권장).
- 미등록 교재 요청함 서브탭에는 대기 중 요청 개수 배지 표시 (기존 요청함 스펙 유지).
- 톱니바퀴 아이콘에도 대기 중 요청이 있으면 개수 배지 표시 (메인 화면에서도 새 요청이 온 걸 알 수 있게).

## 4. 보안 요구사항

- 화면 전환은 프론트에서 일어나지만, **관리 기능의 모든 서버 함수(요청 처리, 권한 부여/회수, 로그 조회, 접속 현황 조회)는 호출 시점마다 서버 단에서 관리자 권한을 재검증**할 것.
- 프론트에 관리자 화면 HTML/데이터를 미리 내려보내지 말 것 — 관리자 화면 데이터는 톱니바퀴 클릭 후 필요 시점에 로드.

## 5. 기존 기능 영향 범위

- 조회 탭 3개의 로직(공용 관/공통 월 포함, 미등록 표시, 등록 요청 버튼)은 변경 없음.
- 직원·뷰어의 화면은 변화 없어야 함 (원래 관리용 탭이 없었으므로).
- 변경 후 확인 사항: 관리자로 접속 → 톱니바퀴 진입 → 4개 서브탭 정상 동작 → 복귀 시 조회 상태 유지 / 직원·뷰어로 접속 → 톱니바퀴 미노출 + 관리 서버 함수 직접 호출 시 거부되는지 확인.

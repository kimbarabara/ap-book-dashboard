# 교재 대시보드 — 권한 체계 정비 및 기능 추가 요구사항

## 1. 현재 시스템 개요 (변경하지 말 것, 참고용)

- **탭 3개** (관리자/직원/뷰어 공통 노출):
  1. **교재 낱개 검색**: 교재명 또는 ISBN 일부 입력 → 카드 형태로 결과 표시
  2. **반별 교재 조회**: 관 / 반(학년) / 사용 월 모두 선택 후 조회
     - `공용` 관으로 등록된 교재는 어떤 관을 선택해도 항상 포함
     - `공통` 월로 등록된 교재는 어떤 월을 선택해도 항상 포함
     - 교재DB에 없는 교재명은 "교재DB 미등록"으로 표시 (에러로 중단되지 않음)
  3. **과목별 교재 조회** (※ 기존 이름 "관-학년-과목 조회"에서 변경 — UI 탭 라벨을 "과목별 교재 조회"로 수정할 것. 기능은 그대로 유지)
- PDF 저장은 브라우저 인쇄 기능 사용
- 기존 관리자 전용 기능: 체크박스 + 실시간 "★ 청구 총액", 접속자 현황, 교재DB 시트 연결 버튼
- 권한 등급: **관리자 / 직원 / 뷰어** (뷰어 = 알바생, 강사)

### 보안 원칙 (필수)
- **가격 데이터는 서버(백엔드) 응답에서부터 차단할 것.** 뷰어 권한의 요청에는 단가·총액 필드를 응답 데이터에 아예 포함하지 않는다. 프론트에서 CSS/JS로 숨기는 방식 금지 (개발자 도구로 노출됨).
- 권한 체크는 모든 서버 함수(엔드포인트) 진입 시점에 수행한다. 프론트 UI 분기만으로 처리하지 않는다.
- 뷰어의 PDF는 별도 처리 불필요 — 화면 자체에 가격이 없으므로 인쇄해도 가격 없는 목록이 출력됨.

## 3. 신규 기능 요구사항

### 3-1. 교재 등록 요청 (직원·뷰어 → 관리자)

**목적**: "교재DB 미등록"이 뜰 때 그냥 넘어가지 않고 데이터가 정리되는 선순환 만들기. 아직 편성표에 없는 신규 교재도 미리 요청 가능.

요청 방법은 두 가지 유형이 있으며, 저장되는 요청 데이터 구조와 이후 처리 흐름(요청함)은 동일하다.

**유형 A — 미등록 표시 기반 요청** (조회 결과에서 발견한 경우)
- 반별/과목별 교재 조회 결과에서 "교재DB 미등록"으로 표시된 교재 옆에 **[등록 요청]** 버튼 표시 (직원·뷰어 모두).
- 버튼 클릭 시 간단한 요청 폼(모달):
  - 교재명: 미등록으로 표시된 이름 **자동 입력** (수정 가능)
  - 선택 입력: 출판사, ISBN, 메모 (예: "표지에는 이렇게 적혀 있음")
  - 필수 입력은 교재명뿐. 입력 부담 최소화.
- 발견 위치(관/반/월)는 조회 중이던 조건으로 **자동 기록**.

**유형 B — 자유 입력 요청** (편성표에도 아직 없는 신규 교재를 미리 요청)
- 진입점: 헤더(또는 교재 낱개 검색 탭)에 **[신규 교재 등록 요청]** 버튼 (직원·뷰어 모두 노출).
- 요청 폼은 유형 A와 동일 + 추가 필드:
  - "사용 예정 반" (관/반/월 선택, **선택 입력**) — 유형 A의 자동 기록을 대신함
- **중복 방지 필수 단계**: 교재명 입력 후 제출 전에, 입력값으로 교재DB를 자동 검색(기존 낱개 검색 로직 재사용)하여 유사한 교재가 있으면 "혹시 이 교재 아닌가요?" 목록을 먼저 표시. 사용자가 "아니요, 새 교재입니다"를 선택해야 제출 가능.
  - 처리 대기 중인 요청 목록도 같은 방식으로 검사하여 유사 요청이 있으면 표시.

**두 유형 공통**
- 요청 저장 시 자동 기록: 요청자 이메일(계정), 요청 시각, 요청 유형(A/B), 발견 위치 또는 사용 예정 반.
- **중복 요청 방지**: 동일 교재명으로 처리 대기 중인 요청이 있으면 버튼 대신 "이미 요청됨 (처리 대기 중)" 배지 표시, 버튼 비활성화.
- **반려 안내**: 요청이 반려된 경우, 해당 요청자가 대시보드를 다시 열었을 때 "요청하신 ○○ 교재: 반려 (사유)" 표시. (승인된 경우는 미등록 표시가 사라지거나 검색에 나타나는 것으로 충분, 별도 알림 불필요)

### 3-2. 미등록 교재 요청함 (관리자)

- 관리자 화면에 **대기 중 요청 개수 배지** 표시 (예: 요청함 🔔 3).
- 요청함 목록: 교재명, 요청 유형(미등록 발견/신규 요청), 요청자, 요청 시각, 발견 위치 또는 사용 예정 반, 메모, 상태(대기/처리됨/반려).
- 요청 클릭 → **교재DB 등록 폼이 요청 정보로 미리 채워진 상태로 열림** → 등록 완료 시 요청 상태 자동 "처리됨" 변경.
- 반려 처리 시 반려 사유 입력 필드 제공.

### 3-3. 권한 관리 화면 (관리자)

- 대시보드 안에서 계정 권한을 직접 관리하는 UI. (시트/코드를 직접 수정하지 않아도 되도록)
- 기능:
  - 이메일 입력 → 권한(관리자/직원/뷰어) 부여
  - 기존 계정 권한 변경 / 회수(삭제)
  - 계정 목록에 **등록일** 표시 (알바생 교체가 잦아 퇴사자 계정 회수 누락 방지 목적)
- 데이터는 권한 시트(또는 테이블) 기반: 이메일 | 권한 | 이름(선택) | 등록일

### 3-4. 변경 이력 / 감사 로그 (관리자 조회 전용)

- **교재DB 수정**(등록/수정/삭제)이 발생할 때마다 자동 기록: 시각 | 수행자 이메일 | 동작(등록/수정/삭제) | 대상 교재 | 변경 내용 요약 (수정인 경우 변경 전→후 값)
- 권한 변경(부여/회수)도 동일하게 로그에 기록.
- 관리자 화면에서 로그 조회 가능 (최신순, 간단한 필터: 기간/수행자).
- 목적: 청구 금액이 걸린 데이터이므로 분쟁 방지용 근거 확보.

### 3-5. "이 교재를 쓰는 반" 역조회 (전 권한 공통)

- 교재 낱개 검색 탭의 결과 **카드 클릭(또는 카드 내 버튼)** 시, 해당 교재가 등록된 **관 / 반 / 사용 월 목록** 표시.
- 용도: "이 책 어느 반이 쓰죠?" 문의에 즉답.
- 기존 반별 교재 데이터를 역방향으로 조회하면 되므로 별도 데이터 구조 추가 불필요.
- 뷰어에게도 노출하되, 역조회 결과에도 가격 정보는 포함하지 않음 (2번 보안 원칙 동일 적용).

### 3-6. 업데이트 공지 (전 권한 공통 조회, 작성은 자동 + 관리자 수동)

**목적**: 대시보드에 어떤 기능이 추가/변경됐는지, 어떤 데이터가 반영됐는지 사용자(직원·뷰어 포함)가 확인할 수 있게 함.
**감사 로그(3-4)와의 구분**: 감사 로그는 관리자용 상세 데이터 변경 추적 기록이고, 업데이트 공지는 전체 사용자 대상 요약 소식. 별개 기능임.

- **저장소**: 교재DB 스프레드시트에 `업데이트` 시트를 신설하고, 웹앱은 이 시트를 읽어서 표시.

**공지 생성 방식 — 3가지 (자동 2 + 수동 1)**

1. **데이터 변경 자동 요약** (완전 자동):
   - 웹앱을 통한 교재DB 변경(등록/수정/삭제) 발생 시, 감사 로그 기록과 함께 업데이트 시트용 요약을 생성.
   - **하루 단위로 1~3행 요약** (건별 개별 공지 금지 — 스팸 방지). 변경 성격별로 행을 나눌 수 있음 (예: 신규 등록 1행, 단가 수정 1행, 삭제 1행). 같은 날 같은 성격의 추가 변경 발생 시 해당 행을 갱신.
   - **신규 등록 행은 등록된 교재명을 전부 나열** ("외 N건"으로 줄이지 말 것). 예: `교재 3건 신규 등록: A Summer to Die, The Giver, Longman Academic Reading 4`
   - 단가 수정·삭제 행은 건수 중심으로 간결하게 표기 가능.
   - 예: `07/17 | 데이터반영 | 교재 2건 신규 등록: A Summer to Die, The Giver | Y`
   - **[보안 필수] 자동 요약에 단가 금액을 절대 포함하지 말 것.** 업데이트 공지는 뷰어도 조회하므로 "단가 N건 수정"처럼 건수만 표기. 교재명은 표기 가능(가격 아님).
2. **기능 업데이트 반자동** (배포 시 자동 반영):
   - 코드 내에 CHANGELOG 상수(배열: 버전 또는 날짜 | 분류 | 내용)를 유지.
   - 웹앱 로드 시 CHANGELOG에 있으나 업데이트 시트에 없는 항목을 발견하면 자동으로 시트에 추가 (중복 방지: 버전/날짜+내용 기준).
   - **[개발 규칙] 이후 이 프로젝트의 코드를 수정하는 모든 작업(Claude Code 포함)은 사용자에게 보이는 변경이 있을 경우 CHANGELOG에 한 줄 항목을 추가할 것.** (내부 리팩토링 등 사용자에게 보이지 않는 변경은 제외)
3. **수동 공지**: 관리자가 시트에 직접 행 추가 (예: "3월 교재 주문 마감 안내"). 이 경로도 계속 동작해야 함.

**표시 UI**
- 헤더(모든 탭 공통 영역)에 "업데이트 소식" 버튼 → 클릭 시 목록 모달 (최신순).
- 최근 7일 이내 새 항목이 있으면 버튼에 개수 배지(NEW) 표시.
- 각 항목: 날짜, 분류 태그, 내용. 분류별 태그 색상 구분 (기능추가/기능수정/데이터반영/공지).
- **표시여부 컬럼**이 `Y`인 행만 노출 (자동 생성 행은 기본 `Y`, 관리자가 시트에서 `N`으로 바꾸면 숨김).

## 4. 데이터 구조 (신규 시트/테이블 제안)

1. **권한 시트**: `이메일 | 권한(admin/staff/viewer) | 이름 | 등록일`
2. **등록요청 시트**: `요청ID | 요청유형(미등록발견/신규요청) | 교재명 | 출판사 | ISBN | 메모 | 요청자이메일 | 요청시각 | 발견위치또는사용예정반(관/반/월) | 상태(대기/처리됨/반려) | 반려사유 | 처리시각 | 처리자`
3. **감사로그 시트**: `시각 | 수행자이메일 | 동작유형 | 대상 | 변경내용요약`
4. **업데이트 시트** (교재DB 스프레드시트 내 신설): `날짜 | 분류(기능추가/기능수정/데이터반영/공지) | 내용 | 표시여부(Y/N) | 생성구분(자동/수동)`

## 5. 구현 순서 제안

1. 서버 단 권한 체크 공통 함수 + 가격 필드 차단 (보안 기반)
2. 미등록 교재 등록 요청 + 요청함 (핵심 신규 기능)
3. 권한 관리 화면
4. 역조회
5. 감사 로그
6. 업데이트 공지 (업데이트 시트 + 표시 UI)

각 단계 완료 시 기존 기능(세 탭 조회, 공용 관/공통 월 포함 로직, 미등록 표시)이 깨지지 않았는지 확인할 것.
